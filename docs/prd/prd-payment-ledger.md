# PRD: 支付平台核心账务与结算系统

## Problem Statement

当前跨境收款平台仅支持"收款→换汇→提现到国内"的单一业务流。升级为全球钱包平台后，需要一套完整的复式记账系统，覆盖收单、收款、换汇、提现、保证金管理等全场景的资金流转、余额计算、退款校验和对账能力。

现有系统缺乏：
1. 多币种钱包余额管理（available / pending / frozen_hold / reserve）
2. 收单场景下的分层清算逻辑（T+2 上游结算 vs T+7 商户结算）
3. 退款校验与资损防护机制
4. 固定保证金与滚动保证金的并存管理
5. 负余额处理与自动抵扣机制
6. 流水账与账本的双层记录体系

## Solution

构建一套基于复式记账的支付核心系统，包含：
- **账本层**：不可变的双重记账分录，作为余额计算的唯一真相源
- **流水账层**：业务事件记录，与账本通过 reference_id 关联
- **余额层**：缓存余额 + 乐观锁 + 定期对账，支持 O(1) 查询
- **保证金层**：固定保证金与滚动保证金并存，支持累计、释放、升级、退款退回
- **退款层**：四种 Case 的退款校验与记账逻辑，支持负余额自动抵扣

## User Stories

### 账户与余额

1. 作为平台，我希望为每个客户创建独立的账户体系（available / pending / frozen_hold / reserve:fixed / reserve:rolling），以便物理隔离客户资金
2. 作为平台，我希望账户名中包含客户 ID（`customer:{id}:{type}:{ccy}`），以便直接从账户名追溯客户归属
3. 作为平台，我希望每个币种有独立的余额，以便支持 USD、EUR、GBP 等多币种管理
4. 作为平台，我希望使用乐观锁（lock_version）控制并发更新，以便防止余额计算冲突
5. 作为平台，我希望缓存余额并在写入时同步更新，以便支持 O(1) 余额查询
6. 作为平台，我希望定期从账本分录重新计算余额并与缓存对比，以便发现并修复余额漂移

### 收单

7. 作为商户，我希望每笔收单交易在 Capture 时记录到 pending 余额，以便看到待结算资金
8. 作为平台，我希望在 T+7 结算时扣除 MDR、滚动保证金、固定保证金后将净额入 available，以便完成商户结算
9. 作为平台，我希望上游结算（T+2）与商户结算（T+7）独立记账，以便分离平台成本与商户结算
10. 作为平台，我希望支持冻结 pending 资金到 frozen_hold，以便风控拦截可疑交易
11. 作为平台，我希望冻结和解冻操作都有复式记账分录，以便审计追溯

### 退款

12. 作为商户，我希望待结算交易退款时直接扣减 pending 余额，以便实时看到退款影响
13. 作为商户，我希望已结算交易退款时从 available 余额扣减，以便使用已到账资金
14. 作为商户，我希望已结算交易退款时同步退回滚动保证金（仅 HELD 状态），以便拿回对应的保证金
15. 作为平台，我希望固定保证金不因单笔退款而调整，因为它是全局目标金额
16. 作为平台，我希望已释放或已升级的滚动保证金不退回，因为资金已不在 rolling 账户中
17. 作为平台，我希望允许退款产生 available 负余额，以便保障买家退款权益
18. 作为平台，我希望负余额从后续收入（新结算 + 保证金释放）自动抵扣，以便追回平台敞口
19. 作为平台，我希望退款校验包含金额校验、状态校验、窗口校验、幂等校验，以便防止资损

### 保证金

20. 作为平台，我希望同一笔交易可以同时扣固定保证金和滚动保证金，以便灵活组合风控策略
21. 作为平台，我希望固定保证金按原单百分比扣除直到目标金额封顶，以便控制商户风险敞口
22. 作为平台，我希望滚动保证金按原单百分比扣除且不封顶，以便优先保障保证金覆盖
23. 作为平台，我希望扣除顺序为 MDR → 滚动保证金 → 固定保证金，以便滚动保证金优先扣足
24. 作为平台，我希望保证金比率与 MDR 独立配置不做交叉校验，以便解耦配置管理
25. 作为平台，我希望固定保证金支持从 available 余额一次性扣除、从结算中逐笔扣除、商户充值三种累计方式，以便灵活满足不同商户需求
26. 作为平台，我希望滚动保证金在 N 天后自动释放到 available，以便商户按计划回收资金
27. 作为平台，我希望支持滚动保证金全额升级为固定保证金（直接转，不经过 available），以便商户风险等级下降时简化管理
28. 作为平台，我希望升级后将原滚动保证金 entries 状态改为 RESERVE_RELEASED，以便退款时正确判断是否退回
29. 作为平台，我希望固定保证金手动释放到 available，以便在商户关闭账户或降级时操作

### 流水账

30. 作为运营，我希望查询每笔交易的完整生命周期（Auth → Capture → Settle → Refund），以便处理商户和买家的查询
31. 作为财务，我希望流水账与账本通过 reference_id 关联，以便从任何一方追溯到另一方
32. 作为风控，我希望查询每次冻结/解冻的操作记录（操作人、时间、原因），以便合规审计

### 风控

33. 作为平台，我希望监控商户连续负余额情况并告警，以便及时干预
34. 作为平台，我希望商户负余额超过阈值时冻结出金，以便控制平台风险敞口

## Implementation Decisions

### 模块划分

系统由以下核心模块组成：

1. **Ledger Service（账本服务）**
   - 负责所有复式记账分录的写入
   - 分录不可变，错误通过反向分录冲正
   - 每笔分录包含 entry_id、account、direction、amount、currency、reference_id

2. **Balance Service（余额服务）**
   - 维护缓存余额（available / pending / frozen_hold / reserve:fixed / reserve:rolling）
   - 写入时同步更新缓存，使用乐观锁防并发
   - 定期从账本重新计算并与缓存对账

3. **Settlement Service（结算服务）**
   - 管理 T+7 商户结算批次
   - 扣除顺序：MDR → 滚动保证金 → 固定保证金
   - 固定保证金封顶保护，滚动保证金不封顶

4. **Refund Service（退款服务）**
   - 四种 Case 的退款校验与执行
   - 退款来源由原交易状态和保证金类型共同决定
   - 允许已结算交易退款产生负余额

5. **Reserve Service（保证金服务）**
   - 固定保证金：累计（结算扣除/充值）、释放、目标管理
   - 滚动保证金：累计、定时释放、全额升级
   - reserve_entries 表逐条记录，附 release_date

6. **Transaction Log Service（流水账服务）**
   - 记录交易、退款、结算、资金流动、账户活动、操作审计六类流水
   - 与账本通过 reference_id 关联

### 账户结构

```
客户账户（Liability）:
  customer:{id}:available:{ccy}
  customer:{id}:pending:{ccy}
  customer:{id}:frozen_hold:{ccy}
  customer:{id}:reserve:fixed:{ccy}
  customer:{id}:reserve:rolling:{ccy}
  customer:{id}:special_account:{ccy}

平台账户:
  house:bank:{ccy}           (Asset)
  receivable:txn:{ccy}       (Asset)
  revenue:fee:acquiring      (Revenue)
  expense:refund             (Expense)
  expense:card_network_fee   (Expense)
  expense:acquirer_fee       (Expense)
```

### 上下游独立

- 上游（Acquirer → PF）：T+2 结算，记录通道费成本
- 下游（PF → 商户）：T+7 结算，基于原单金额计算 MDR + 保证金
- 两本账独立，互不影响

### 保证金扣除顺序与逻辑

```
原单金额 → MDR → 滚动保证金 → 固定保证金 → 商户实收

固定保证金:
  实际扣除 = min(原单金额 × 比率, 目标差额, 剩余余额)
  封顶保护，不会导致负余额

滚动保证金:
  实际扣除 = 原单金额 × 比率
  不封顶，可能导致负余额
```

### 退款校验规则

```
规则 1: 退款金额 ≤ 原交易金额 - 已退款累计
规则 2: 交易状态为 CAPTURED 或 SETTLED
规则 3: 在退款窗口期内（卡支付 180 天）
规则 4: 仅 Case 1（待结算）需要 pending ≥ 退款金额
规则 5: refund_id 唯一（幂等）

Case 2/3 允许 available 为负（不拒绝）
```

### 退款保证金退回

| Case | 原交易状态 | 保证金类型 | 保证金状态 | 退回 |
|------|-----------|-----------|-----------|------|
| 1 | 待结算 | — | — | 无 |
| 2 | 已结算 | 滚动 | HELD | ✅ 按比例 |
| 2b | 已结算 | 滚动 | RELEASED / RESERVE_RELEASED | ❌ |
| 3 | 已结算 | 固定 | — | ❌ |

## Testing Decisions

### 测试原则

- 只测试外部行为，不测试实现细节
- 每个场景覆盖完整的资金流向（余额变化）
- 幂等性测试：相同操作重复执行，结果不变
- 并发测试：多线程同时操作同一账户，余额一致

### 测试模块

1. **账本分录测试**
   - 每种业务场景的分录正确性
   - 借贷平衡校验
   - 不可变性（分录写入后不可修改）

2. **余额计算测试**
   - 缓存余额与账本一致性
   - 乐观锁冲突重试
   - 负余额抵扣逻辑

3. **退款测试**
   - 四种 Case 的退款分录
   - 保证金退回（HELD / RELEASED / RESERVE_RELEASED）
   - 负余额产生与抵扣
   - 幂等性（重复退款不重复执行）

4. **保证金测试**
   - 固定保证金累计与封顶
   - 滚动保证金累计与自动释放
   - 升级（全额、直接转、entries 状态更新）
   - 同时扣两种保证金
   - 充值专户累计

5. **结算测试**
   - T+7 结算批次处理
   - 扣除顺序：MDR → 滚动 → 固定
   - 退款后的结算金额计算

6. **对账测试**
   - 缓存余额 vs 账本余额
   - reserve_entries 余额 vs reserve 账户余额

## Out of Scope

1. KYC/AML/制裁筛查等合规模块
2. 上游收单行通道对接
3. 银行账户开户与管理
4. 多币种换汇的银行通道对接
5. 前端 UI / 商户后台
6. API 网关与认证授权
7. 收款（Collection）和提现（Payout）的通道对接
8. 公司卡（Corporate Card）业务
9. 支付 API 对外开放

## Further Notes

### 业界参考

- Adyen：每笔支付约 50 条账本记录，使用模板数学验证借贷平衡
- Stripe：内部账本每天处理约 50 亿条事件
- Square：专门构建 Books 服务——不可变的复式记账数据库
- 所有主流支付公司均采用复式记账

### 关键设计约束

1. 账本分录不可变，错误通过反向分录冲正
2. 余额从分录推导，缓存仅为性能优化
3. 流水账与账本独立，通过 reference_id 关联
4. 保证金比率与 MDR 解耦，不做交叉校验
5. 固定保证金封顶保护，滚动保证金不封顶
